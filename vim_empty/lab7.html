Polymorphism

When editing text, every change can fundamentally be categorized as either an insertion or a deletion. To support tracking of changes (and subsequently undoing and redoing), we will employ polymorphism. We want to ultimately add every change to a list, then on undo or redo, call the appropriate function without the need to consider whether it was a deletion or insertion. This can be accomplished by creating a Change interface that Insertion and Deletion classes will implement. When we instantiate a Change object, we essentially decide at creation how it should act whenever e.g. undo() is called on this Change. To create an interface, start by creating the base class, in this example, Change. Decide what components are constant among all Change's, for example the text that changed, and implement that like you would any other class. Then for the parts that depend on the implementation we will make virtual. This tells the compiler to defer definition details to any child class. In the end, it will look something like 
    /* Class provides interface for children */
    class MyInterface {
        int allObjectsHaveThis;             /* To be defined by MyInterface */
        virtual void functionToDo();        /* Children must implement this */
    };

    /* Must implement the interface left by parent class */
    class MyInterfaceChild : public MyInterface {
        void functionToDo() { /* Do something */ };
    };
Note that there can be a function definition in the parent class that is the "default" if the children don't implement it.

Optimizing Current Code

We are going to revisit the code from last week to make some optimizations. The first is going to fix the egregious storage of a file in one C string. It is extremely bulky, and at some point, there won't be one contiguous block of memory large enough to hold the file. Fix your internal storage to instead hold an array of C strings (char**), where each string contained is a line of the file. How you decide to do this is up to you, but a good strategy might be to originally malloc enough space for the entire line, or half of it, and if you must realloc, double the length. That said, you need to keep track of the length of lines. One possible way of doing this is to encode it into the first four bytes of the string (typecasting will be your friend), and when writing to ncurses, start at the fourth byte. It may also be useful to keep a pointer in your Document class to the place in your array that the ncurses cursor is so you know where new characters go. Now that we have this better system in place, if the cursor moves out of the window, reprint everything to give the appearance of scrolling. If your window is 20 lines, you originally show lines 0-19 of the file. If your cursor is on the bottom line and it moves down, you should then show lines 1-20, leaving the cursor on the bottom line. To help facilitate times when you must rewrite the entire display, you should add to your Document class a refresh(line, column) that will clear your display and then fill it with the top left character being at the argued line and column in the file.

Using this, we can make the editing section faster by only reprinting characters that change. If a character was inserted (other than a newline), rewrite only characters in that line, shifting right those after the new character. If a new line is present, only rewrite the contents below it, pushing down everything below the newline. The new line should now contain everything that was in the same line to the right of the new character.

Finally, we will slightly improve saving, but more to come next week. For now, the best we can do is simply not save if we don't have to. Keep track of whether the file has been edited since the last save, and on save, only write to a file if it has been edited OR it is writing to a new file (not the one that was originally opened).

Undo and Redo

We want to start tracking changes so we can undo and redo. The way we will do this is with a doubly-linked list of changes. The reason we do this instead of an array is because we will always have a pointer to the next change to undo, which is chronologically one before the next change to redo. We will only ever access one of those two changes, and so accessing this linked list will always be O(1). We can then take advantage of the O(1) insertion and deletion without worrying about the usual linked-list drawback that indexing is O(n). Implement this functionality such that each vim-keybound edit gets a Change. This means that everything that happens during one continuous INSERT mode goes into one Change, as does each edit by shortcut that occurs during DEFAULT mode (e.g. 'x' to delete a character). Every Change should know the line and column number (both 0 indexed) of the first character that changed, as well as a string that contains what changed (either inserted or deleted). To make it a doubly-linked list, it will also need a next and prev field that points to other Change's. Finally, every change needs some notion of how to undo and redo, but this depends on the type of change, so these will be virtual void undo() = NULL and virutal void redo() = NULL. Create Insertion and Deletion classes that inherit from Change and implement undo() and redo(), where Insertion::undo() deletes the inserted code (which is stored in Change along with its location in the file). Update your Vim class to contain a (smart) pointer to the Change corresponding to the next undo() call. Every time there is a change to the document, a new Change should be added to your linked list and your nextUndo should point to that most recent change. In doing this, you lose any Change's that you could have redone because you have to reset the next pointer. You may need a dummy Change when this list is empty to avoid ambiguity when you've undone every Change, knowing whether there is anything you can redo.

New Keybindings

Again, we will have required keybindings to exhibit functionality and optional ones for enrichment.
    Required:
    • all required keybindings from last week should still work
    • u - undo the last change if in DEFAULT mode. If no more changes, do nothing.
    • . (period) - redo the next change if in DEFAULT mode. If no more changes to redo, do nothing.
    • $ - in DEFAULT mode, move cursor to the last character of the line.
    • ^ - in DEFAULT mode, move cursor to the first character of the line.
    Optional:
    • b - in DEFAULT mode, moves the cursor to the first character of the current word (previous word if already on the first character) in the file.
    • e - in DEFAULT mode, moves the cursor to the last character of the current word (next word if already on the last character) in the file.
    • w (lower case) - in DEFAULT mode, moves the cursor to the first character of the next word in the file.
    • / (forward slash) - when in DEFAULT mode, enters forward search mode. Anything typed after the slash and before an <ENTER> will be searched. On <ENTER>, the cursor moves to the next instance of that string in the file, wrapping if necessary. The cursor does not move if the string is not found. In this mode, the bottom ncurses line should display / <string_to_search>. The searched string should be stored somewhere for future reference, as should the search direction.
    • ? (question mark) - when in DEFAULT mode, enters reverse search mode. This does the same thing as forward search except it moves the cursor to the beginning of the first instance BEFORE the current cursor location. In this mode, the bottom line should display ? <string_to_search>.
    • n - when in DEFAULT mode, recalls the most recently searched string and direction and finds the next instance of it in the appropriate direction.
